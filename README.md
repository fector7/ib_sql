## Задание 1. PostgreSQL Authentication

### **1. Какие методы аутентификации используются для подключения по TCP/IP с адресов 127.0.0.1/32 и ::1/128?**

`trust`

```
host all all 127.0.0.1/32    trust
host all all ::1/128         trust
```

---

### **2. Какие методы аутентификации используются для подключения по TCP/IP со всех остальных адресов, кроме указанных в предыдущем пункте по протоколу?**

`md5`

```
host all all all md5
```

### **3. Верно ли утверждение: пароль роли app хранится в виде функция_хеширования password (пароль хранится в поле rolpassword)? Если не верно, то приведите описание алгоритма, который используется для хранения хеша.**

Нет, утверждение неверно.

Используется алгоритм MD5: хеш вычисляется от конкатенации пароля и имени пользователя — `md5(пароль || имя_роли)`, результат сохраняется с префиксом `md5`.

Фактическое значение из `pg_authid` для роли `app` с паролем `pass`:
```
md50d06d8eb2a5f36ab73d83481186543d5
```
Проверка: `md5('pass' + 'app')` = `0d06d8eb2a5f36ab73d83481186543d5` ✓

### **4. Какое значение имеют поля rolsuper, rolcreaterole, rolcreatedb, rolbypassrls с указанием назначения данных столбцов? t будет означать «да», f — «нет». См. https://postgrespro.ru/docs/postgresql/12/catalog-pg-authid.**

Для роли `app`:

| Поле | Значение | Назначение столбца |
|------|----------|--------------------|
| `rolsuper` | t | Является суперпользователем — обходит все проверки прав доступа |
| `rolcreaterole` | t | Может создавать другие роли |
| `rolcreatedb` | t | Может создавать новые базы данных |
| `rolbypassrls` | t | Игнорирует политики безопасности на уровне строк (Row-Level Security) |

### **5. Почему значения полей rolcanlogin и rolpassword для роли app не изменились, и вы по-прежнему можете подключиться с помощью psql без указания пароля, хотя в pg_hba.conf для host all all all указано reject?**

Поля `rolcanlogin` и `rolpassword` хранятся в системном каталоге `pg_authid` и не имеют отношения к методу аутентификации в `pg_hba.conf`. Переменная `POSTGRES_HOST_AUTH_METHOD=reject` влияет только на генерацию строк в `pg_hba.conf` при первичной инициализации контейнера — она не изменяет существующие записи каталога ролей. Поэтому пароль и флаг `rolcanlogin` роли `app` остаются неизменными.

Подключение через `docker-compose exec postgres psql` работает без пароля, потому что клиент запускается внутри контейнера и использует **Unix domain socket**, который соответствует правилу `local` в `pg_hba.conf`:

```
local   all   all                 trust   ← совпадает первым (socket)
host    all   all   127.0.0.1/32  trust
host    all   all   ::1/128       trust
host    all   all   all           reject  ← до этого правила не доходит
```

Правила читаются сверху вниз, останавливаясь на первом совпадении. `local all all trust` совпадает раньше, поэтому `reject` для `host` к локальному подключению не применяется.

## Задание 2. CIS PostgreSQL Benchmarks

### **1. Какие методы не рекомендуется использовать для удалённых подключений?**

- `trust`
- `password`
- `ident`
- `md5`

### **2. Какие методы рекомендуется использовать для удалённых подключений?**

- `scram-sha-256`
- `cert`

> Из CIS PostgreSQL 16 Benchmark v1.1.0:  
> *"The trust, password, and ident authentication methods should not be used for remote connections. The scram-sha-256 method should be used instead of md5. Certificate (cert) authentication is also acceptable for remote connections."*
